# Курсова работа по Програмиране на C
## Задача 6.07.3 — Външно сортиране на цели числа

## Условие

Да се сортират в нарастващ ред цели числа, които се четат от зададен файл.
Буферът, в който се четат числата, е ограничен до N позиции (задава се при
стартиране на програмата). Сортираният файл да се запише като нов
(да не се унищожава оригиналът).

## Идея на решението

Понеже буферът е ограничен до N, не мога просто да заредя всичко в паметта
и да викна qsort. Затова направих External Merge Sort на 2 фази:

1. Чета по N числа, сортирам ги в паметта и ги записвам в малки временни
   файлове (run_0.tmp, run_1.tmp, ...).
2. Сливам всички тези малки сортирани файлове в един голям сортиран файл,
   като на всяка стъпка взимам най-малкото число измежду главите им.

## Компилация (Windows)

Трябва ми GCC. Най-лесно: сваляне на MinGW-w64 от https://winlibs.com/,
разархивиране в `C:\mingw64\` и добавяне на `C:\mingw64\bin` в PATH.

Проверка: отварям нов CMD и пиша `gcc --version`.

След това в CMD влизам в папката на проекта и пиша:

```cmd
gcc -Wall -O2 -o bin\external_sort.exe src\external_sort.c
gcc -Wall -O2 -o bin\generate.exe      src\generate.c
```

### Какво прави `gcc`

GCC е компилатор — превежда моя C код на машинен код (.exe), който
процесорът разбира директно. Преводът минава през 4 стъпки:
препроцесор (разрешава #include), компилатор (C → асемблер),
асемблер (асемблер → машинен код) и линкер (свързва с printf, malloc).

Флаговете:
- `-Wall` — показва всички предупреждения, за да хваща глупави грешки
- `-O2` — оптимизация ниво 2, прави програмата осезаемо по-бърза
- `-o bin\external_sort.exe` — име и място на готовия .exe
- `src\external_sort.c` — входният .c файл

Ако не искам да помня флагове, минимумът е:
```cmd
gcc src\external_sort.c -o bin\external_sort.exe
```

## Стартиране

```cmd
bin\external_sort.exe <вход> <изход> <N> [-v]
```

- `<вход>` — файлът с числата
- `<изход>` — името на новия сортиран файл
- `<N>` — размер на буфера (брой числа едновременно в паметта)
- `-v` — verbose, печата всяка стъпка (само за малки файлове)

Генератор на тестови файлове:
```cmd
bin\generate.exe <изходен_файл> <брой_числа>
```

## Измерени времена

| Брой числа | N (буфер) | Парчета | Време |
|------------|-----------|---------|-------|
| 1 000      | 50        | 20      | <0.01 с |
| 10 000     | 1 000     | 10      | 0.015 с |
| 100 000    | 10 000    | 10      | 0.10 с |
| 1 000 000  | 100 000   | 10      | 0.93 с |
| 1 000 000  | 1 000     | 1000    | 4.5 с |
| 1 000 000  | 500 000   | 2       | 0.6 с |
| 10 000 000 | 500 000   | 20      | 9.08 с |

Ако N е твърде малко → много парчета → бавна фаза 2.
Ако N е голямо → малко парчета, но нарушаваме духа на условието.
В практиката добро правило е N около корен от броя числа.

## Как работи програмата в детайли

Пример: вход `42 7 13 99 1 56 23 8 77 4 31 65 12 3 88` (15 числа), N=4.

### Фаза 1 — Разделяне

Чета по 4 числа, сортирам ги в паметта с qsort и записвам всеки блок
като отделен файл:

```
Итерация 1: чета [42, 7, 13, 99]  → qsort → [7, 13, 42, 99]  → run_0.tmp
Итерация 2: чета [1, 56, 23, 8]   → qsort → [1, 8, 23, 56]   → run_1.tmp
Итерация 3: чета [77, 4, 31, 65]  → qsort → [4, 31, 65, 77]  → run_2.tmp
Итерация 4: чета [12, 3, 88]      → qsort → [3, 12, 88]      → run_3.tmp
Итерация 5: fscanf не успява      → break
```

Накрая имам 4 малки сортирани файла на диска.

### Фаза 2 — Сливане (k-way merge)

Идеята: ако имам k сортирани списъка, най-малкото общо число задължително
е едно от първите им елементи. Защо? Защото всеки списък е сортиран —
значи първият му елемент е минимумът В НЕГО. Глобалният минимум тогава
е минимумът сред тези k минимума.

Поддържам 3 паралелни масива с дължина k:
- `runs[i]`  — указател към отворения файл i
- `heads[i]` — последно прочетеното число от файл i
- `alive[i]` — 1 ако файлът има още числа, 0 ако е свършил

Подготовка: отварям всичките 4 файла и чета по едно число от всеки.

```
heads = [7, 1, 4, 3]
alive = [1, 1, 1, 1]
```

Главен цикъл — на всяка стъпка намирам индекса с най-малката глава,
записвам това число в изхода, и чета следващото от същия файл:

```c
while (1) {
    int min_idx = -1;
    long min_val;
    for (int i = 0; i < run_count; i++) {
        if (alive[i] && (min_idx == -1 || heads[i] < min_val)) {
            min_val = heads[i];
            min_idx = i;
        }
    }
    if (min_idx == -1) break;          // всички парчета свършиха

    fprintf(fout, "%ld\n", min_val);

    if (fscanf(runs[min_idx], "%ld", &heads[min_idx]) != 1) {
        alive[min_idx] = 0;
        fclose(runs[min_idx]);
    }
}
```

Пълно проследяване за нашия пример:

| # | heads | min | от парче | действие |

| 1 | [7, 1, 4, 3]   | 1  | 1 | пиша 1, чета 8 |
| 2 | [7, 8, 4, 3]   | 3  | 3 | пиша 3, чета 12 |
| 3 | [7, 8, 4, 12]  | 4  | 2 | пиша 4, чета 31 |
| 4 | [7, 8, 31, 12] | 7  | 0 | пиша 7, чета 13 |
| 5 | [13, 8, 31, 12]| 8  | 1 | пиша 8, чета 23 |
| 6 | [13, 23, 31, 12]| 12 | 3 | пиша 12, чета 88 |
| 7 | [13, 23, 31, 88]| 13 | 0 | пиша 13, чета 42 |
| 8 | [42, 23, 31, 88]| 23 | 1 | пиша 23, чета 56 |
| 9 | [42, 56, 31, 88]| 31 | 2 | пиша 31, чета 65 |
| 10| [42, 56, 65, 88]| 42 | 0 | пиша 42, чета 99 |

### Фаза 3 — Почистване

В края изтривам временните файлове с `remove()` и освобождавам паметта
с `free()`. Оригиналът не е пипан, защото съм го отворил с `"r"` (само
четене) и изходът е друг файл.

## Сложност

- Време: O(M log N) за фаза 1 (qsort на блокове) + O(M · k) за фаза 2,
  където M = брой числа, N = размер на буфера, k = брой парчета.
- Памет: O(N + k) — само буферът и масивите за главите.
- Диск: оригиналът + изходът + временни парчета.

## Други варианти, които обмислях

- Naive (qsort на всичко в паметта) — не става, нарушава условието и
  забива при големи файлове.
- k-way merge с min-heap вместо линейно търсене — O(M log k). Полезно
  когато k е голямо (стотици/хиляди), но кодът е значително по-сложен.
  За моите тестове k е до около 100 и линейното търсене е по-просто и
  достатъчно бързо.
- Replacement selection за фаза 1 — генерира по-малко на брой парчета
  с двойна средна дължина. По-сложно за писане, малка печалба тук.
- Polyphase merge — стара техника от епохата на лентите, остаряла днес.
- Radix sort — би работил, но кодът става много по-сложен.

Избрах External Merge Sort, защото е прост, четлив, точно отговаря на
условието и работи за произволно големи файлове.
